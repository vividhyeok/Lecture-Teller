# LectureTeller 리뷰 및 재설계 제안

작성일: 2026-03-19

## 1. 한 줄 결론

LectureTeller는 `STT를 직접 잘하는 앱`도 아니고 `내부 AI 자동화 앱`도 아니다.  
가장 맞는 정체성은 `외부에서 만든 강의 텍스트와 외부 AI 결과물을 가장 적은 마찰로 학습용 TTS 자산으로 바꾸는 워크플로 앱`이다.

또 하나 중요한 점은 `1패널이냐 3패널이냐`가 본질이 아니라는 것이다.  
기존 3패널은 실제로 한 흐름의 작업을 애매하게 갈라서 불편했고, 그래서 1패널로 줄이려 한 판단은 자연스러웠다. 다만 지금은 반대로 너무 많이 한 곳에 접어 넣어서 흐름이 죽었다.  
즉 문제는 패널 수가 아니라 `작업 흐름의 연결성`이다.

## 2. 이번에 실제로 확인한 범위

- 백엔드: `app.py`, `migrate.py`, `requirements.txt`, `start.bat`, `run.bat`
- 구 UI: `static/index.html`, `static/main.js`, `static/style.css`
- React UI: `lectureteller-react/src/*`, `package.json`, `README.md`
- 데이터/설정: `data.json`, `data_v2.json`, `settings.json`, `.gitignore`, 루트 `README.md`

- `npm run build` 통과

## 3. 이 앱의 목표를 다시 명확히 정의

아래 6개 원칙이 현재 프로젝트의 가장 맞는 기준점이다.

1. LectureTeller는 STT를 직접 잘하려고 하기보다, 외부에서 생성된 강의 텍스트를 받아 학습용 TTS 자산으로 바꾸는 앱으로 설계한다.
2. 강의 음성의 STT는 NotebookLM에서 처리하고, 앱 내부에서는 교수별 프롬프트 프로필과 최종 프롬프트 조립 기능에 집중한다.
3. 교수별 프로필에는 반복 설명 성향, PPT 낭독 비중, 핵심 개념 압축 방식, 원하는 설명 난이도 같은 규칙을 저장한다.
4. 사용자는 STT 본문과 PDF 내용을 넣고 교수 프로필 및 출력 타입을 선택하면, Claude 같은 외부 AI에 바로 복붙할 수 있는 완성 프롬프트를 얻는다.
5. 외부 AI가 생성한 최종 대본은 다시 LectureTeller로 가져와 버전별로 저장하고, 기존 TTS 기능으로 음성을 생성해 반복 학습에 활용한다.
6. 이 앱의 핵심은 내부 AI 자동화가 아니라, 외부 AI 결과를 가장 적은 마찰로 학습 가능한 오디오와 자료로 전환하는 워크플로를 만드는 것이다.

## 4. 현재 상태 요약

- v1 정적 UI와 v2 React UI가 동시에 살아 있다.
- 루트 `/`는 여전히 v1 HTML을 서빙하고, `/v2`에만 React UI가 올라간다.
- 데이터도 `data.json`과 `data_v2.json`으로 이중화되어 있다.
- 실제 데이터 기준으로 v1은 과목 3개 / 단원 17개인데, v2는 과목 1개 / 클립 1개만 있다.
- README는 React SPA + 3패널 구조를 설명하지만, 실제 `App.tsx`는 `PlayerPanel`만 렌더링한다.
- 즉 지금은 "옛 구조의 흔적"과 "중간에 급하게 1패널로 좁힌 흔적"이 동시에 남아 있는 상태다.

## 5. 가장 큰 문제들

### A. 목적은 맞는데, 핵심 흐름이 1급 기능으로 승격되지 않음

- 현재 코드도 TTS와 플레이어 중심이라는 점에서는 방향이 완전히 틀린 것은 아니다.
- 하지만 지금 앱이 가장 잘해야 하는 흐름은 `STT/PDF 입력 -> 교수 프로필 선택 -> 프롬프트 조립 -> 외부 AI 결과 붙여넣기 -> 버전 저장 -> TTS -> 반복 학습`이다.
- 현재는 이 전체 흐름보다 `과목/주차/클립 관리`, `오디오 재생`, `주변 UX 기능`이 앞서 있다.

### B. 3패널을 버린 이유는 타당했지만, 대체 구조가 완성되지 않음

- 기존 `Explore / Compose / Player` 구조는 실제로 하나의 작업을 부자연스럽게 나누는 면이 있었다.
- 그래서 1패널로 좁히려 한 건 좋은 판단이다.
- 다만 현재 `App.tsx`는 `PlayerPanel`만 렌더링하고, 예전 상태와 컴포넌트는 그대로 남아 있어 구조가 더 혼란스러워졌다.
- 즉 `3패널이 나빴다 -> 그래서 1패널로 간다`까지는 맞았는데, `1패널 안에서 어떤 순서로 무엇을 하게 할 것인가`가 아직 안 잡혔다.

### C. v1 / v2 / 데이터 소스 / 문서가 분리되어 있음

- `app.py:68-73`, `app.py:444` 기준으로 v1과 v2가 동시에 노출된다.
- `static/main.js`는 `/api/library`를 쓰고, React는 `/api/v2/library`를 쓴다.
- 루트 `README.md`는 v2를 메인처럼 설명하지만 루트 진입점은 아직 v1이다.
- 이 상태에서는 어떤 UX를 고쳐도 기준 화면이 계속 흔들린다.

### D. 외부 AI 워크플로가 아직 메인 기능이 아님

- 현재 `PromptModal`은 외부 AI 활용을 위한 힌트 수준이다.
- 하지만 네 목적 기준에서는 이것이 보조 기능이 아니라 핵심 흐름의 일부여야 한다.
- 특히 부족한 것은 아래다.
- 교수별 프로필 저장
- 출력 타입별 프롬프트 조립
- STT + PDF + 프로필 + 출력 타입을 합친 최종 프롬프트 생성
- 외부 AI 결과 대본 버전 저장

### E. 내부 AI 요약 기능은 현재 철학과 충돌함

- `PlayerPanel`과 `app.py`에는 내부 OpenAI 요약 생성 흐름이 있다.
- 하지만 현재 제품 철학은 "좋은 결과는 외부 AI에서 직접 뽑고, 앱은 그 결과를 활용한다"에 가깝다.
- 이 기준에서는 내부 요약 기능은 메인이 아니라 선택 옵션 또는 후순위다.
- 지금처럼 전면에 섞여 있으면 제품 정체성이 흐려진다.

### F. 버전 관리 모델이 약함

- 현재 `과목 > 주차 > 클립` 모델은 오디오 보관함으로는 쓸 수 있지만,
- `원문`, `교수 스타일별 설명본`, `시험 직전 버전`, `짧은 복습 버전`, `Q&A 버전`을 한 세션 안에서 관리하기엔 부족하다.
- 핵심은 "클립 몇 개를 모으는 것"이 아니라 "같은 강의에 대한 여러 결과 대본과 오디오 버전을 추적하는 것"이다.

## 7. 기획 리뷰: 기능 축소보다 재배치가 더 중요함

### 대체로 필요한데 배치가 비효율적인 것

- 과목 / 주차 / 파일 구조 자체
- 플레이어 주변 편의 기능
- 연속재생, 문장 점프, 요약 뷰, 드래그 정렬
- 긴 입력 처리, 버전 관리, 반복 학습 보조
- 탭/사이드바/모달/스타일 시스템

즉, 지금의 문제는 "기능이 너무 많다"보다 "필요한 기능들이 핵심 흐름에 맞게 정렬되어 있지 않다"에 가깝다.

### 부족한 것

- NotebookLM에서 만든 STT를 넣는 공식 입력 경로
- 교수별 프롬프트 프로필 저장
- STT + PDF + 프로필 + 출력 타입을 합친 최종 프롬프트 조립
- 외부 AI 결과 대본을 버전별로 저장하는 흐름
- 결과 대본별 TTS 재생성
- 같은 강의에 대한 여러 결과 버전 비교

### 없느니만 못한 것

- 애매하게 쪼개진 3패널 구조
- 반대로 모든 흐름을 아무 구획 없이 한 패널에 밀어 넣는 구조
- 내부 AI 요약과 외부 AI 활용 흐름의 중복
- `prompt/confirm/alert/window 전역 함수` 혼용

## 8. 추천 제품 정체성

### 정의

`외부 STT + 외부 AI 결과를 학습용 TTS 자산으로 바꾸는 반자동 워크플로 앱`

### 핵심 역할

- 강의 텍스트와 PDF를 정리한다.
- 교수별 설명 스타일을 프로필로 저장한다.
- 외부 AI에 넣을 최종 프롬프트를 조립한다.
- 외부 AI가 만든 결과 대본을 버전별로 저장한다.
- 그 대본으로 TTS를 만든다.
- 반복 학습에 맞는 플레이어 경험을 제공한다.

### 하지 않아도 되는 것

- STT를 직접 최고 수준으로 처리하려고 애쓰기
- 내부 LLM 자동화를 메인 가치로 밀기
- 모든 단계를 앱 안에서 완전 자동화하려고 하기

## 10. 패널 구조에 대한 추천

### 결론

1패널도 가능하고, 다중 패널도 가능하다.  
중요한 건 `한 작업이 다음 작업으로 이어질 때 화면 전환 때문에 문맥이 끊기지 않느냐`다.

### 추천 방향

`단일 워크스페이스 + 내부 구획 분리`가 가장 적합하다.

즉 바깥에서는 1패널이지만, 내부는 아래처럼 나누는 방식이다.

1. 자료 입력 구역
2. 교수 프로필/출력 타입 선택 구역
3. 최종 프롬프트 조립 구역
4. 외부 AI 결과 대본 버전 구역
5. TTS/플레이어 구역

이렇게 하면 기존 3패널처럼 억지로 나뉘지도 않고, 현재처럼 모든 걸 한 카드에 욱여넣지도 않게 된다.

### 넓은 화면에서는 이렇게 확장 가능

- 좌측: 강의 목록 + 교수 프로필
- 중앙: STT/PDF + 프롬프트 + 결과 대본
- 우측: TTS 버전 + 플레이어

즉 `1패널 철학`을 유지하면서도 실제로는 `잘 나뉜 다중 구역`을 가질 수 있다.

## 11. 가장 추천하는 제품 흐름

### 1단계. 자료 입력

- NotebookLM에서 만든 STT 텍스트 붙여넣기
- PDF 텍스트 또는 PDF 메모 입력
- 강의명 / 세션명 지정

### 2단계. 교수 프로필 선택

- 교수별 프로필 선택
- 출력 타입 선택
- 예시 출력 타입:
- 쉬운 설명본
- 정리본
- 시험 직전 암기본
- Q&A형 설명본

### 3단계. 최종 프롬프트 조립

- STT 본문
- PDF 내용
- 교수 프로필 규칙
- 출력 타입 규칙
- 추가 사용자 메모

위 요소를 합쳐 Claude 등 외부 AI에 바로 넣을 수 있는 완성 프롬프트를 만든다.

### 4단계. 결과 대본 수입

- 외부 AI 결과를 앱에 다시 붙여넣는다.
- `summary`, `explain`, `cram`, `qa` 등 타입별로 저장한다.
- 같은 세션 안에 여러 버전을 저장한다.

### 5단계. TTS 생성 및 반복 학습

- 선택한 버전에서 TTS 생성
- 음성별로 여러 오디오 저장 가능
- 자주 듣는 버전 고정
- 반복 재생, 연속 청취, 빠른 버전 전환

## 12. 기존 장점은 무엇을 살릴까

## 14. UX 재설계 방향

### 지금보다 더 단순해야 한다

- 지금처럼 `탐색`, `작성`, `재생`을 화면 수준에서 분리하면 실제 작업 흐름이 끊긴다.
- 반대로 모든 것을 플레이어 카드 하나에 넣으면 작업 의도가 흐려진다.
- 따라서 추천은 `한 작업 화면 안에서 단계별 블록이 이어지는 구조`다.

### 추천 메인 화면

- 상단: 강의 세션 컨텍스트
- 좌측: 라이브러리 + 교수 프로필
- 중앙 상단: STT/PDF 입력
- 중앙 중단: 프롬프트 조립
- 중앙 하단: 외부 AI 결과 대본 버전
- 우측 또는 하단: TTS/플레이어

### 생성/삭제 UX

- `prompt/confirm/alert` 제거
- 입력은 인라인 폼 또는 정식 모달
- 실패 메시지는 toast 또는 인라인 에러로 통일

## 15. 기술 리팩터링 제안

### 백엔드

- `app.py`를 라우터/서비스/스토리지 계층으로 분리
- v1 API 제거 또는 `legacy`로 격리
- JSON 저장소를 유지하더라도 새 스키마 기준으로 명확히 정의
- 교수 프로필 / 프롬프트 템플릿 / 스크립트 버전 / 오디오 버전 API 분리

### 프론트엔드

- `App.tsx`를 `단일 워크스페이스` 기준으로 재구성
- 죽은 `activeTab` 구조 제거
- 설정/모달 제어를 store 기반으로 통일
- `Sidebar`의 생성/삭제를 정식 UI 플로우로 교체
- `PromptModal`은 단순 보조가 아니라 메인 워크플로 일부로 승격 또는 별도 블록화
- CSS를 토큰/레이아웃/컴포넌트 레벨로 재정리

## 17. 최종 권고

- 이 프로젝트는 `무엇을 자동화할까`보다 `어디까지 외부 AI에 맡기고, 앱은 어디서 마찰을 줄일까`를 기준으로 재설계하는 것이 맞다.
- 네 개발 스타일 기준에서 가장 좋은 방향은 `완전 자동화`가 아니라 `반자동이지만 가장 높은 품질로 가장 빠르게 학습 자산을 만드는 구조`다.
- 따라서 가장 추천하는 방향은 아래다.
- v1 완전 종료
- v2 메인 고정
- 3패널 잔재 제거
- 그렇다고 무조건 1덩어리로 몰지 말고, 단일 워크스페이스 내부 구획으로 재구성
- 교수 프로필 + 프롬프트 조립 + 결과 대본 버전 관리 + TTS 재생성을 메인 기능으로 승격
- 플레이어는 화려함보다 학습 효율 중심으로 재정렬

이 방향으로 가면 지금의 "과하게 프로그래밍된 느낌"을 줄이면서도, 기존에 이미 만들어 둔 TTS/플레이어 자산의 장점은 살리고, 네 목적에 가장 잘 맞는 앱으로 수렴할 수 있다.
